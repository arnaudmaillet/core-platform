# Staging provisioning runbook (B1 — hybrid managed backends)

Staging runs the fleet on **managed MSK + ElastiCache** with **in-cluster
scylla-operator + CNPG**, its **own ArgoCD**, and **External Secrets** for creds.
Everything below is authored + statically validated (terraform validate, helm
template, kubectl kustomize, kubeconform) but **has not been applied** — there's
no live cluster access from the repo. Apply in this order.

## 0. Prereqs
- AWS creds for the target account (724772065879, us-east-1).
- A GitHub token for the Terragrunt `github_repository_file` (writes global-params-staging.json).

## 1. Terragrunt — infra (in dependency order)
```
cd infrastructure/live/staging/us-east-1
terragrunt apply --working-dir networking/vpc
terragrunt apply --working-dir eks
terragrunt apply --working-dir data/msk                 # -> bootstrap brokers + AmazonMSK_…_app secret
terragrunt apply --working-dir data/elasticache         # -> config endpoint + …-redis-auth secret
terragrunt apply --working-dir data/opensearch          # -> search domain + …-opensearch-master secret
terragrunt apply --working-dir data/media-bucket        # -> media S3 bucket
terragrunt apply --working-dir data/audit-kms           # -> audit KEK
terragrunt apply --working-dir data/audit-worm          # -> audit Object-Lock bucket (needs audit-kms)
# IRSA scopes audit/media app roles to the buckets+KEK above, so it runs AFTER them.
terragrunt apply --working-dir security/irsa-roles      # ESO role + audit/media app roles
terragrunt apply --working-dir kubernetes/argocd        # staging ArgoCD; writes global-params-staging.json
```
> Ordering matters: `security/irsa-roles` now consumes `data/audit-kms`,
> `data/audit-worm`, `data/media-bucket` outputs (mock_outputs let a dry `plan`
> run first). `terragrunt run-all apply` resolves this DAG automatically.
The MSK SCRAM + Redis AUTH credentials are **generated by Terraform** into Secrets
Manager (no manual seeding). Capture the outputs:
```
terragrunt output --working-dir data/msk bootstrap_brokers_sasl_scram
terragrunt output --working-dir data/elasticache configuration_endpoint
```

## 2. Fill the env endpoint placeholders
The data-store endpoints are dynamic, so the staging `<svc>.env` files (and the
KEDA scaler patches) carry placeholders. Replace before (or as part of) the
ArgoCD sync:
- `<<MSK_BOOTSTRAP_BROKERS_SASL_SCRAM>>` → the MSK output above (in the `.env`s
  **and** `patch-{counter-worker,realtime-dispatcher,audit-worker}-scaler.yaml`)
- `<<ELASTICACHE_CONFIG_ENDPOINT>>` → the ElastiCache output above
- `<<OPENSEARCH_ENDPOINT>>` → `terragrunt output --working-dir data/opensearch endpoint` (search.env)
- `<<KEYCLOAK_TOKEN_ENDPOINT>>` → the Keycloak token URL (auth.env; see §2b)
- `<<AUTH_JWKS_URL>>` → auth-server's JWKS endpoint (realtime.env; see §2b)
- `<<ACM_CERTIFICATE_ARN>>` → the issued ACM cert ARN for `*.core-platform.click`
  (`realtime-gateway-public.yaml`, NLB TLS termination). The public WSS endpoint
  resolves at `realtime-staging.core-platform.click` (external-dns). Confirm the
  ACM cert covers that single-level host (a `*.core-platform.click` wildcard does).

(Recommended follow-up: have the Terragrunt data components write these into a
ConfigMap the overlay references, the same way the argocd module writes
global-params — so this step disappears.)

## 2a. Seed the application secrets (out-of-band → Secrets Manager)
ESO reads `core-platform-staging-*` from Secrets Manager. The managed-backend
creds (MSK/Redis/OpenSearch master) are written by Terraform, but these app
secrets must be created manually (the ExternalSecrets reference them):
- `core-platform-staging-media-s3` `{access_key, secret_key}` — an IAM user's
  static keys. ⚠ The media/audit object clients use **rusty-s3 with static keys**,
  NOT the IRSA role on their SA — so an IAM user (or static keys) is required even
  though Block 4 created IRSA roles. The IRSA roles are harmless and become live
  if/when the code moves to the AWS SDK credential chain.
- `core-platform-staging-audit-crypto` `{object_access_key, object_secret_key,
  witness_access_key, witness_secret_key, kek_base64, signing_key_base64}` — audit
  static S3 keys + the v1 ENV-KEK + checkpoint signing key. ⚠ Real AWS KMS + a
  cross-account WORM witness remain the documented external deferral; this wires
  the v1 path so audit can run, with the same WORM bucket used for the witness.
- `core-platform-staging-auth-secrets` `{signing_private_pem, signing_public_pem,
  keycloak_client_secret}` — ES256 keypair + Keycloak client secret.

## 2b. Keycloak (auth prerequisite — not yet provisioned)
auth brokers to a federated Keycloak that is **not** stood up by this repo. Until
it is: provide its token endpoint (`<<KEYCLOAK_TOKEN_ENDPOINT>>`), the client
secret (above), and auth-server's JWKS URL for realtime (`<<AUTH_JWKS_URL>>`).
realtime's WS plane fails closed (RTM-1001) until the first JWKS fetch succeeds;
its gRPC health plane is unaffected, so the pod still goes Ready.

## 3. GitOps brings up the cluster
Staging's ArgoCD (from step 1) bootstraps from `infrastructure/argocd/bootstrap/staging`:
- **operators** appset installs scylla-operator + external-secrets + **keda**
  (+ cnpg, k6) at sync-wave -10, with the ESO ServiceAccount annotated with its
  IRSA role from global-params-staging. KEDA must converge before the workload
  overlay (wave 0) so the keda.sh CRDs exist when the ScaledObjects apply.
  > Confirm each worker's MSK topic has ≥ its ScaledObject `maxReplicaCount`
  > partitions (counter-worker=12, realtime-dispatcher=8, audit-worker=4) — a
  > consumer group cannot scale past its partition count.
- **security/platform** appsets install cert-manager, lb-controller, etc.
- **workloads** appset (`deployments/staging/fleet.yaml`) syncs `k8s/overlays/staging`.

## 4. Apply the ScyllaCluster (after the operator is healthy)
```
kubectl apply -k k8s/base/infra/scylla-cluster   # ns scylla; creates scylla-client
```

## 5. Verify
- `kubectl -n external-secrets get externalsecret backend-creds` → SecretSynced
- `kubectl get secret backend-creds` → KAFKA_SASL_*/REDIS_PASSWORD present
- ScyllaCluster `scylla` Ready (3 members); CNPG `staging-account-postgres` Ready
- Services: gRPC readiness SERVING (migrator init ran, backends reachable)
